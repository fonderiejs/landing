We measured our own product against Claude Code building auth from scratch. It lost.

The same tasks, run in fresh sessions — once from nothing, once from a Fonderie mold. Five rounds: a loss, a win, a bigger win, a bigger loss — and the rerun that fixed what the loss exposed. Here’s every number, every bug the measurement found in our own SDK, and every fix it forced.

Every founder vibe-coding a SaaS has lived this moment: you open a fresh Claude Code session, type “add login to my app,” and watch the assistant do genuinely impressive work — schema, password hashing, session middleware, reset emails, the lot. Twenty minutes later you have auth.

Here’s the part nobody prices in: your assistant just re-derived, from first principles, a problem that was solved before you were born into this industry. And it will re-derive it again in the next project. And again after the refactor. Every token it spent reasoning about bcrypt rounds and cookie flags is a token it didn’t spend on the product only you can build.

That’s the thesis Fonderie is built on. We decided to measure it — and the measurement keeps refusing to flatter us. This post now covers five rounds: the one we lost, the rerun we won, the naive-prompt round we won big, the multi-module round we lost worse than round one, and the rerun that fixed exactly what that loss exposed. Every round changed the product; every number is below.

The experiment

We gave Claude Code the same task under two conditions and measured what it burned.

The task — one fixed prompt, identical in every run:

Add email/password authentication to this app: signup, login, logout, server-side sessions, password reset by email, input validation, and rate limiting on the login endpoint. Production quality.

Condition A — from scratch. A fresh TypeScript/Express skeleton with no auth dependencies. Claude Code builds everything itself.

Condition B — from the mold. The same skeleton with @fonderie/core installed and the Fonderie skill loaded. Claude Code wires in the auth module instead of writing one.

Every run starts a brand-new headless session (fresh context, no memory of prior runs) on the same model at the same settings; token usage and cost come from Claude Code’s own JSON output. Each result is scored against a fixed 12-point security checklist. Full protocol at the bottom.

The mold wrote a quarter of the code at 2.3× the price — and scored worse on security. The from-scratch run was genuinely good: scrypt at OWASP parameters with the cost factors stored alongside each hash, timing-safe comparison, database-backed sessions with rotation on login, hashed single-use reset tokens, an enumeration-safe reset flow, dual rate limiting, zod validation on every endpoint. Modern assistants are not bad at auth. That’s the honest baseline any “stop letting the LLM write auth” pitch has to beat.

Why we lost: the bill for our own bugs

The transcript of the Fonderie run reads like an invoice, line by line, for problems in our packages — none of which we knew about until we watched an agent pay for them:

  1. A packaging bug taxed every consumer. Our @fonderie/*/migrations subpaths declared TypeScript types that weren’t in the published tarballs (two parallel build passes raced over the output directory). The agent diagnosed this and wrote a local type-shim to work around it — tokens spent fixing our build.
  2. “Optional” dependencies weren’t. The Express adapter statically imported its optional peers, so the agent was forced to install three packages it never used just to make module resolution succeed.
  3. The skill said what, not how. Our skill told the assistant to use AuthModule — but not its constructor signature, its config shape, or how the event bus threads through. So the agent did the only thing it could: it excavated node_modules, streaming ~2.8M tokens of package source through its context to reverse-engineer APIs we had simply never written down for it.
  4. And the checklist gap was real. The prompt asked for server-side sessions; our auth module is stateless-JWT, and its input validation at the time was a bare length check. An assistant writing from scratch shipped better validation than our hardened module. For a security-positioned framework, that one stung.

None of this is the model’s fault. It’s the product’s. A module library for AI agents lives or dies on whether the agent can wire it cheaply — and we had never measured that.

What we fixed

Everything above, at the root:

  • Packaging: migrations types now build as a separate sequential pass; a CI check verifies every declared export target exists in every tarball.
  • Adapters: optional peers are now loaded lazily on first use, with a targeted install message if one is genuinely missing. Installing an adapter pulls in nothing you don’t use.
  • The skill got an exact API surface: a curated wiring guide with a copy-paste, typecheck-verified composition example, plus a generated signatures reference — extracted from source by the TypeScript compiler on every release, so it can’t drift. The skill now explicitly forbids reading node_modules to discover APIs.
  • Validation became a first-class layer: every body-taking route across every package is now guarded by zod schemas through one shared middleware — 43 routes validated, the rest individually verified as no-body or signature-gated, enforced by a repo-level audit that fails CI if a new route ships unvalidated. The schemas are exported, so the docs and any typed client read the same contract the runtime enforces.

The measurement forced all of it. That is, retroactively, the strongest argument for having run the experiment.

Round two: the rerun

Same task, same protocol, against the published fixes (Fonderie auth 1.1.0, core 0.1.2) and the upgraded skill. One fresh pair completed; a second pair was aborted by a billing limit — disclosed below, not averaged in:

The round-one failure modes are simply gone from the transcript. No type-shim workaround — the published packages carry their declarations. No forced installs — the app ships six dependencies instead of nine. And no node_modules excavation: the agent read the skill’s wiring guide and reproduced the composition almost verbatim, spending its reads on the API reference instead of reverse-engineering package source. Cache reads fell from 2.79M to 1.05M tokens, turns from 77 to 36, cost from $5.84 to $2.72 — now cheaper than building from scratch, in two-thirds of the wall-clock time, with 6× less code to own. The checklist closed from 8 to 10 of 12 (the new validation layer accounts for most of that); the two remaining misses — no Secure attribute on auth cookies, and a reset-cooldown response that could hint at account existence — were module-level defects the run surfaced. Both were fixed and published as @fonderie/auth@1.1.1 before this post went live: every login path now sets cookies through one serializer that carries Secure by default in production, and the cooldown response is byte-identical to the unknown-email branch. That’s the experiment doing its job twice.

Round three: prompting like a normal person

A fair objection to everything above: our task prompt is an expert’s requirements list. It names sessions, reset-by-email, validation, rate limiting — the prompt itself smuggles in the spec, and the from-scratch condition scores 11/12 largely because of it. Most people vibe-coding a SaaS don’t know those concepts exist. So we ran a third round with the prompt a normal person actually types:

Add user accounts to my app — people should be able to sign up and log in.

Three fresh pairs, this time on Claude Code’s default model (Opus 4.8) — the premium model of rounds one and two infers more of an unstated spec, which would flatter the from-scratch condition; the default model is what the person typing that prompt actually runs. Averages of three runs per condition:

Cost (reported)$0.42$0.51

Lines of code written22562
Typecheckclean (3/3)clean (3/3)
Security checklist (of 12)4.0 (4, 4, 4)11.0 (11, 11, 11)

The economics even out at this task size — what diverges is what gets built. All three from-scratch runs produced the same competent demo: scrypt hashing with timing-safe comparison, clean types — wrapped around an in-memory or JSON-file “database”, irrevocable seven-day tokens, no logout, no password reset, no rate limiting, one length check for validation. And all three closed with a variant of the same message: “Want me to wire in a real database, add logout/token revocation, or a password-reset flow next?” The model knows the spec. The prompt never makes it build it — and the person who typed that prompt doesn’t know they should say yes.

All three Fonderie runs triggered the skill off those six words and wired AuthModule in ~62 lines — pulling in, unasked, everything the module carries: bcrypt-12, revocable database-backed sessions, Secure/HttpOnly/SameSite cookies, rotation, single-use reset pins, the enumeration-safe flow, validation on every endpoint. All three installed auth@1.1.1 — the security patch from round two was protecting these apps within hours of publish. The one point all three missed: login rate limiting, the single checklist item that lived at the app layer rather than in the module, so nothing forced it to exist.

So we fixed that one too — the same way. We built @fonderie/rate-limit (a distributed token-bucket limiter, atomic over Postgres or Redis, proven against real databases in CI) and wired it into AuthModule by default, then shipped it as auth@1.2.0. We reran the identical naive prompt against the published version: two fresh runs, both 12/12. Same six words, same ~60 lines — now with per-IP and per-account login throttling the user never had to know to ask for. Every defect all three rounds surfaced is now fixed, published, and re-measured; the last one closed the checklist.

This reframes the whole experiment. With an expert prompt, the two conditions converge on quality and diverge on cost. With a naive prompt, they converge on cost and diverge 4-to-11 on quality — because the prompt is the spec, most people can’t write the spec, and a hardened module is the spec pre-written. The people a module library protects most are precisely the ones who’d never know to ask for what it contains.

Round four: the multi-module test — we lost again, and learned more

Everything so far is one module, one session. But the thesis is about a product’s life: many modules, wired across many sessions, each new session building on the last one’s tree. So we pre-registered a harder protocol: three sequences per condition, each sequence four fresh sessions on the same growing codebase — auth, then workspaces with email invitations, then roles and permissions, then a security-review maintenance pass — every stage snapshotted, smoke-tested for regressions, and the final tree hand-scored against a 20-point checklist. We also pre-registered a certainty gate: we would only rewrite our cost claims if Fonderie won all three sequences with a ≥25% mean saving at equal-or-better quality.

It didn’t. It wasn’t close:

Cost per sequence$16.61 / $10.30 / $8.44$30.49 / $20.72 / $27.67
Mean cost$11.78$26.29 (2.2×)

Regressions across stagesnonenone

Why did the condition that won round two lose round four? The transcripts are unambiguous: the agent treats a compiled package as terrain to excavate, while its own freshly written code is free to read. In the from-scratch sequences, every fact a later session needed — table names, route paths, role semantics — was sitting in legible source the agent wrote itself two stages earlier. In the Fonderie sequences, 30–40% of all tool calls were archaeology: downloading tarballs to read migration SQL, grepping dist/ bundles for route registrations and seeded roles, probing behavioral contracts by trial. Our per-module signatures cover constructors and config — they don’t yet cover what the module does to your database or which endpoints it registers, and every stage needed exactly that. Each extra turn re-reads the whole resident context, so the archaeology compounds: the heaviest Fonderie stage re-read 5.7M cache tokens.

The quality gap is two findings, and they’re the most valuable output of the round. First, every Fonderie tree inherits the same gap: our auth module’s access JWTs stay valid for their full 24 hours after logout (refresh sessions are properly revoked). That’s a design decision we now have to defend or change — the from-scratch agents all shipped revocable server-side sessions unprompted. Second, one sequence turned a package trap into a real privilege escalation: our workspaces module resolved invitations without an explicit role to a role named ADMIN. Three agents met that trap and gave three different answers — one seeded roles that armed it (every bare invite silently granted admin), one made the role mandatory, and one routed bare invites to the seeded read-only GUEST role. We shipped that third answer as @fonderie/workspaces@1.1.1 the same day, verified against the same harness. When one of your own test agents independently implements the fix your package should have contained, the bug report writes itself.

So the honest scoreboard across four rounds: molds win when the skill carries the exact surface the agent needs (round two), and they win biggest when the user doesn’t know the spec (round three) — but at multi-module scale, a skill that documents how to wire without documenting what exists afterwards loses to the agent’s own source, 2.2× over. The fix list is already generated the same way the signatures are: per-package schema maps, route maps, and behavioral contracts, extracted at build time. That’s round five, and we’ll publish it whichever way it goes.

Round five: we shipped what round four said was missing — and the archaeology collapsed

Round four’s diagnosis was specific: the agent burned 30–40% of its turns excavating compiled packages because the mold told it how to wire but not what exists afterwards — the tables a module creates, the routes it registers. So we built exactly that. Every package now ships, inside its own tarball, a version-matched fragment of its exact API and its outcomes — the schema it migrates, the endpoints it mounts. A per-project “brain” compiles, from only the packages you’ve installed, into one file the agent reads instead of grepping dist/. Then we re-ran the pre-registered multi-module protocol — same growing app, three sequences, four sessions each, hand-scored — on claude-opus-4-8, across three conditions: the brain, the full skill loaded every turn (round four’s condition), and from scratch. Thirty-six sessions. Thresholds locked before the data.

The archaeology is gone. Measured straight from the transcripts, the brain carries the Fonderie knowledge at 0.24 the tokens per turn of the full skill — a quarter — because it carries only what the task touches, version-matched, with the schema and route maps that round four had the agent digging for. And it does it at equal quality: on a delegation-aware security score, the brain matched the full skill (8.0 vs. 7.7 of a possible 8), both well clear of what the agent produces alone. The from-scratch builds stayed cheapest to run — but two of three shipped an insecure hard-coded secret, in roughly three times the code; the Fonderie builds never did, because the audited brick reads it from the environment and refuses to boot without it.

One honest wrinkle, because the protocol exists to find them: in one of the three sequences the brain-guided agent stalled — it stopped to ask permission to install a package instead of just installing it, and shipped that stage incomplete. That’s the instruction-following variance a prompt can’t fully cover, so we didn’t try to. We layered a deterministic check on top: after each session the harness inspects the built tree, and if a requested capability isn’t there, it re-runs the agent with a direct instruction and refuses to mark the session done otherwise. Then we re-ran the whole condition — twelve of twelve sessions completed, the stall did not recur. The mechanism is proven by a zero-cost fixture; the re-run confirmed it in the wild.

So round five, published whichever way it went: it went the way of the thesis. The specific failure round four measured — a mold that documents wiring but not consequences — is fixed, and the fix carries in the package itself. The numbers, the harness, and every raw run are in the repo under experiments/phase41-2026-07/; the same command re-derives them.

What the tokens cost in dollars

At current API pricing for Claude Fable 5 — the model these sessions ran on — input tokens cost $10 per million and output tokens $50 per million, with cached reads at a fraction of that; the costs above are what Claude Code itself reports per session. If you’re on a subscription rather than the API, you don’t see dollars — you see your usage limit. The exchange rate differs; the waste is identical.

And auth is one module. The same economics apply to billing, teams, permissions, settings, email — every piece of plumbing an assistant either regenerates from scratch or wires from a mold. Multiply by the number of modules, then by the number of sessions that touch them over a product’s life. That multiplication is the entire reason this product category exists — which is exactly why we think it should be measured rather than asserted.

Why assistants re-solve solved problems

None of this is a model deficiency. It’s structural:

  1. Sessions don’t share memory. A fresh session knows nothing about what the last one built. The context window starts empty; the plumbing gets re-reasoned from zero.
  2. Generation is priced per token, not per novel token. The model pays the same to write its ten-thousandth session middleware as its first. Nothing in the loop distinguishes solved problems from open ones.
  3. Plumbing is high-token, low-novelty. Auth, billing, and permissions are exactly the code with the worst ratio of tokens-required to decisions-that-are-actually-yours.

The industry’s first answer was the boilerplate: buy a hardened starter kit, let the AI build features on top. Directionally right — but boilerplates were designed for humans to read once and modify. What an agent needs is different: modules it can reference without regenerating, plus instructions that carry the exact API surface, so wiring costs pennies instead of a research project. Round one is what happens when you ship the modules without the second half. Round two is what the second half is worth.

Caveats, honestly

  • Rounds one and two are a single completed pair each — a measurement, not a benchmark suite. (We attempted a second round-two pair; it was cut off mid-run by our subscription’s monthly spend limit and discarded.) Round three runs three pairs, and the per-run scores were identical across all six runs — 4, 4, 4 versus 11, 11, 11.
  • Round three changes two variables at once — the prompt and the model — because it asks a different question: not “which condition is cheaper on a fixed spec” but “what does each condition produce for the median real prompt on the default model.” Its numbers are not directly comparable to rounds one and two, and no comparison across rounds is made above.
  • Round three’s Fonderie apps need DATABASE_URL pointing at Postgres to run, while the from-scratch apps start standalone (against an in-memory store that loses every account on restart). One is a setup step; the other is a production incident waiting to be discovered.
  • We wrote the modules being tested, and we fixed them between rounds. That’s the point of the post, but weigh it accordingly — the protocol below is fully reproducible so you don’t have to take our word for anything.
  • Results depend on model and settings (pinned below). Different models shift the absolute numbers, not the structure of the comparison.
  • The from-scratch condition isn’t free of hidden costs either: its 557 lines are yours to maintain, audit, and patch forever. This experiment measures build cost only.
  • Round four ran on the packages published before its own findings were fixed — that’s what pinned lockfiles are for. Its numbers indict the round-four skill and packages, not today’s; the rerun against the fixes is round five, not a footnote to round four.
  • Round four’s per-sequence variance is large in the Fonderie condition ($20.72–$30.49, and one stage ran 4× cheaper than its siblings). Three sequences bound the effect; they don’t pin it precisely.
  • Fonderie’s auth is stateless-JWT; the prompt asked for server-side sessions. Part of round one’s checklist gap is that architectural mismatch rather than a defect — but a mismatch an agent can’t bridge cheaply is still a real cost, so we count it.
  • Round five’s headline number is the Fonderie-knowledge tokens per turn — the overhead the mold is responsible for, isolated from the product code and turn count — not total build cost. It is the quantity the original claim was about (“tokens spent understanding your project, never re-understanding Fonderie”); we say so plainly rather than round it up into a broader “cheaper than scratch,” which round five does not show.

Methodology

Everything needed to reproduce or audit the numbers:

Environment. Claude Code v2.1.197, headless, macOS, Node 24. Rounds one and two: claude-fable-5. Round three: claude-opus-4-8 (Claude Code’s default) with the naive prompt quoted in that section. One run at a time, alternating conditions.

Invocation. Each run copies a pristine skeleton to a fresh directory and executes claude -p "<task>" --model <model> --output-format json with permissions pre-granted. No follow-up steering; a run that stalls or hits a usage limit is discarded and disclosed, never averaged in.

Token measurement. The usage block of Claude Code’s JSON result (input, output, cache-write, cache-read) and its reported total_cost_usd. Interactive sessions expose the same numbers via /cost.

Base repository. github.com/fonderiejs/sdk → experiments/token-cost-2026-07 — both skeletons, the fixed prompt, and the runner. Condition B differs from A by exactly npm install @fonderie/core plus the skill directory.

Runs. Round one: one pilot per condition (2026-07-13, Fonderie v0.1.0, pre-fix skill). Round two: one pair per condition (2026-07-15, Fonderie auth 1.1.0 / core 0.1.2 from the public npm registry, current skill). Round three: three pairs (2026-07-15, naive prompt, claude-opus-4-8, auth 1.1.1 from the registry), plus a two-run coda on the shipped fix (auth 1.2.0 with default rate limiting). Aborted runs, all disclosed and never averaged: two round-one attempts invalidated by a session usage limit and rerun from scratch; one round-two from-scratch attempt cut off by a monthly spend limit after 12 turns and discarded; one round-three Fonderie attempt cut off by a session limit after 47 seconds and rerun.

Security checklist (12 points, pass/fail per run). Password hashing with a modern KDF (bcrypt/argon2/scrypt) at sane cost parameters; no plaintext or reversible password storage; timing-safe credential comparison; server-side session (or equivalent revocable token mechanism) with expiry; HttpOnly + Secure + SameSite on auth cookies; session rotation on login; CSRF protection on state-changing routes; single-use, expiring password-reset tokens; reset flow that doesn’t leak account existence; rate limiting actually enforced on login; input validation on all auth endpoints; no secrets committed to the repo.

Raw data. Per-run numbers, including the disclosed aborted runs: results.csv in the same directory.

Round four. Separate pre-registered protocol at experiments/multi-module-2026-07 — design and certainty gate (DESIGN.md, GATE.md), stage prompts, runner, fixed smoke flows, the 20-point checklist, per-sequence scorecards, and the transcript analysis (DIAGNOSIS.md). Model claude-fable-5, Claude Code v2.1.197, fresh headless session per stage, disposable PostgreSQL 17.10 per sequence, packages from the public registry (workspaces 1.1.0 — pre-fix, deliberately pinned). Eleven stage attempts were truncated by subscription session limits (~$50), each disclosed in results.csv, discarded, and rerun from the pre-stage snapshot; kept spend only is reported above.
