How much does it cost when Claude Code builds auth from scratch? We measured.

The same auth task, run repeatedly in fresh sessions — once from nothing, once from a Fonderie mold. Token counts, dollar costs, and a 12-point security checklist, with a fully reproducible protocol.

Every founder vibe-coding a SaaS has lived this moment: you open a fresh Claude Code session, type “add login to my app,” and watch the assistant do genuinely impressive work — schema, password hashing, session middleware, reset emails, the lot. Twenty minutes later you have auth.

Here’s the part nobody prices in: your assistant just re-derived, from first principles, a problem that was solved before you were born into this industry. And it will re-derive it again in the next project. And again after the refactor. Every token it spent reasoning about bcrypt rounds and cookie flags is a token it didn’t spend on the product only you can build.

We kept saying this in the abstract. Then we decided to measure it.

The experiment

We gave Claude Code the same task under two conditions and measured what it burned.

The task — one fixed prompt, identical in every run:

Add email/password authentication to this app: signup, login, logout, server-side sessions, password reset by email, input validation, and rate limiting on the login endpoint. Production quality.

Condition A — from scratch. A fresh TypeScript app skeleton with no auth dependencies. Claude Code builds everything itself.

Condition B — from the mold. The same skeleton with @fonderie/core installed and the Fonderie skill loaded. Claude Code wires in the auth module instead of writing one.

Each condition ran ⟦N⟧ times, each in a brand-new session (fresh context, no memory of prior runs), on the same model at the same settings. After each run we recorded token usage from Claude Code’s /cost report, wall-clock time, and scored the result against a fixed 12-point security checklist. Full protocol at the bottom.

Turning tokens into dollars

At current API pricing for Claude Fable 5 — the model our Claude Code sessions run on — input tokens cost $10 per million and output tokens $50 per million.

That makes the from-scratch build $⟦A_COST⟧ per run, against $⟦B_COST⟧ with Fonderie — a difference of $⟦DIFF_COST⟧, or ⟦MULTIPLE⟧× the cost, every time an assistant builds auth from nothing.

If you’re on a Claude subscription rather than the API, you don’t see dollars — you see your usage limit. The exchange rate is different; the waste is identical. Those ⟦A_TOTAL⟧ tokens come out of the same budget that was supposed to build your actual product this session.

And auth is one module. The same tax applies to billing, teams, permissions, settings, email — every piece of plumbing your assistant regenerates instead of reusing. Multiply by the number of modules, then by the number of sessions that touch them over a product’s life, and “the AI writes it fresh each time” stops being a workflow and starts being a line item.

Why assistants re-solve solved problems

None of this is a model deficiency. It’s structural:

  1. Sessions don’t share memory. A fresh session knows nothing about what the last one built. The context window starts empty; the plumbing gets re-reasoned from zero.
  2. Generation is priced per token, not per novel token. The model pays the same to write its ten-thousandth session middleware as its first. Nothing in the loop distinguishes solved problems from open ones.
  3. Plumbing is high-token, low-novelty. Auth, billing, and permissions are exactly the code with the worst ratio of tokens-required to decisions-that-are-actually-yours. There is no product insight in a password-reset flow. There are, however, a lot of tokens in one.

The industry’s first answer to this was the boilerplate: buy a hardened starter kit, let the AI build features on top. That’s directionally right, and it’s why kits like MakerKit and ShipFast are now marketing “AI-agent readiness.” But boilerplates were designed for humans to read once and modify. What an agent needs is different: modules it can reference without regenerating, plus instructions — a skill — telling it those modules exist, what they guarantee, and how to wire them.

That’s the bet Fonderie makes: seventeen production-tested TypeScript modules and a skill that teaches any assistant — Claude Code natively, Cursor via rules, Codex via AGENTS.md — to reach for the finished part instead of re-casting it. The measurement above is what that bet is worth on a single module, in a single session.

Caveats, honestly

  • ⟦N⟧ runs per condition is a measurement, not a benchmark suite. Variance across runs was ⟦VARIANCE_NOTE⟧.
  • Results depend on model and settings; ours are pinned in the methodology below. Different models will shift the absolute numbers, not the shape of them.
  • Condition B isn’t free: the module and skill still occupy context, and someone (us) paid the engineering cost of hardening the modules once. That’s rather the point — pay for auth once, industry-wide, instead of per-session.
  • We wrote the modules being tested. The protocol below is fully reproducible precisely so you don’t have to take our word for it.

Methodology

Everything needed to reproduce or audit the numbers:

Base repository. ⟦Link to the public repo containing the starting skeleton, tagged at the exact commit used.⟧ Condition B adds one commit: npm install @fonderie/core plus the skill file, nothing else.

Prompt. The exact task text quoted above, pasted as the first and only user message. No follow-up steering; if a run stalled and required intervention, it was recorded as such.

Token measurement. After each run, /cost in the same Claude Code session, recording the input/output token split it reports. (For continuous measurement across many runs, Claude Code’s OpenTelemetry metrics export the same counters — we used /cost for simplicity.) Dollar figures are computed from the token counts at published API rates, so they hold regardless of whether a session runs on API billing or a subscription.

Runs. ⟦N⟧ per condition, fresh session each, alternating A/B to spread any time-of-day variance. No cherry-picking: every completed run is included in the averages; aborted runs (if any) are listed here: ⟦list or “none”⟧.

Security checklist (12 points, pass/fail per run). Password hashing with a modern KDF (bcrypt/argon2/scrypt) at sane cost parameters; no plaintext or reversible password storage; timing-safe credential comparison; server-side session with expiry; HttpOnly + Secure + SameSite on the session cookie; session rotation on login; CSRF protection on state-changing routes; single-use, expiring password-reset tokens; reset flow that doesn’t leak account existence; rate limiting actually enforced on login; input validation on all auth endpoints; no secrets committed to the repo.
